## Objetivo

Adicionar exploração de dados: viewport controlável, zoom por roda, pan e
migração para Pointer Events (mouse + touch). É a versão de maior risco; por
isso as duas cautelas de código são pré-requisitos obrigatórios.

`fixedY` e `gridPinned`. O viewport do usuário é a **quarta** e não pode virar
mais um `if` no meio.

- [x] Regra: **se existe viewport do usuário, ele ganha na entrada da função e

- [x] Confirmar que `makeView` da [v1.5.0](./v1.5.0-streaming-eventos.md) existe;

## Entregue além do escopo original

Vieram junto no diagnóstico de "zoom lento e vazamento nos cantos" — sem
eles a fase entregava a API mas a experiência não passava no _smoke test_
visual:

- **Series clip.** `renderStatic` envolve séries + legenda em
  `ctx.save() / rect(plot) / clip() / restore()`. Sem isso, viewport
  estreito projetava amostras fora do domínio para pixels bem fora do
  plot rect e o Canvas 2D pintava por cima dos labels dos eixos.
- **Auto-Y windowed (`Viewport.yAuto`).** Zoom em X reescala Y para as
  amostras visíveis (não-stacked e stacked, cada eixo independente). Sem
  isso, zoom em um vale mantinha o pico global dominando a escala e o
  usuário sentia que "o zoom não fez nada". Default `true`; passe
  `yAuto: false` para o comportamento antigo. `ChartOpts.yMin`/`yMax`
  ainda ganham.
- **rAF-coalescing de interação.** Sob `autoDraw: false` (default), o
  wheel/pan mutavam estado mas não agendavam repaint até o próximo
  `pointermove`. Novo `interactionRafId` (separado do `rafScheduled` do
  autoDraw) garante 1 draw por frame em bursts de wheel/pointer.
- **PooledView.** Substituiu o `Object.assign(Object.create(prototype), …)`
  por classe reutilizável (`viewSlotRender` + `viewPoolCrosshair`) —
  zero alocação por série por frame no hot path.
- **`Surface.clientRect()` cacheado** com invalidação em window scroll
  (com `capture: true`) e resize. Remove `getBoundingClientRect()` do
  hot path de wheel/pointer.
- **Reclamp automático em streaming.** `append` / `appendBatch` /
  `appendFrame` / `setData` / `setMaxPoints` / `clear` chamam
  `reclampViewportToExtent()` para o viewport não ficar "órfão" quando o
  ring desliza.

## Arquivos afetados

- `src/charts/chart-base.ts` — viewport (`setViewport` / `getViewport` /
  `resetViewport` / `reclampViewportToExtent`), curto-circuito em
  `updateGridDomain`, wheel/pan/pinch (`onWheel` / `onPointerMove` /
  `onPointerDown` / `onPointerUp` / `updatePinchState`), pool de views
  (`PooledView`, `viewSlotRender`, `viewPoolCrosshair`), rAF coalescing
  (`interactionRafId`, `scheduleInteractionFrame`), auto-Y windowed

  rect (`handleWindowLayoutShift`).

- `src/render/line.ts`, `render/area.ts`, `render/scatter.ts`,
  `render/stacked-band.ts` — passam a iterar `[iStart, iEnd]` via

- `src/render/surface.ts` — `clientRect()` cacheado, `invalidateClientRect()`.
- `src/types.ts` — `Viewport.yAuto`, evento `viewportchange` (já em
  `ChartEventMap`).

## Nota de CHANGELOG / semver

`minor` (aditivo):

> Novo: viewport controlável (com auto-Y opcional), zoom por roda
> (normalizado por `deltaMode`), pan por arraste, pinch-to-zoom com
> handoff para pan, e suporte a Pointer Events / touch. Windowed
> rendering, clip do plot, rAF-coalescing de interação e pool de views
> eliminam vazamento visual e janks. Refatoração interna de
> `updateGridDomain` para priorizar o viewport do usuário. Sem impacto na
> API pública existente exceto o comportamento novo de auto-Y em viewport
> (opt-out via `yAuto: false`).
