# v1.9.0 — Render performance: streaming-first draw pipeline

> **Origem:** vision §6.1, §6.3, §14, §15 · addons §1.2 · diagnóstico pós-v1.8.0
> **Semver:** minor. Mudança interna de pipeline; API pública deve permanecer estável.
> **Risco:** **alto** — toca `ChartBase`, `Surface`, camadas de render e invalidação.
> **Prioridade:** bloqueia qualquer expansão de escopo. Esta fase entra antes de bar chart,
> anotações, subpath exports, CSV ou qualquer novo tipo de gráfico.

## Objetivo

Fazer o render incremental da Goro Charts **voar no cenário certo**: streaming operacional,
dados chegando com frequência alta e canvas redesenhando sem reconstruir partes que não mudaram.

Esta fase não é benchmark abstrato, nem corrida contra uPlot, nem OffscreenCanvas. O alvo é o
posicionamento do produto em [`vision.md`](../vision.md):

> Live and large XY data.

O problema atual é conhecido: no streaming, adicionar 1 ponto marca layout sujo e faz o pipeline
reconstruir fundo, grid, eixos, labels, legenda e todas as séries. Isso desperdiça tempo no que
normalmente não muda entre frames.

## Decisão técnica

### Decisão 1 — Otimizar primeiro o Cenário D, não o benchmark estático

Prioridade desta fase:

1. **Cenário D — alta frequência** (`vision.md` §14.4): ingestão a 100 Hz,
   renderização a 30/60 FPS, sem fila infinita.
2. **Cenário B — dashboard** (`vision.md` §14.2): vários gráficos pequenos/médios.
3. Só depois disso olhar para render bruto estático de 100k+ pontos.

Justificativa: o diferencial da Goro Charts não é vencer um benchmark sintético de render estático.
É entregar streaming previsível com API simples. Otimizar o caminho incremental de streaming serve
diretamente o Pilar 1 do produto.

### Decisão 2 — OffscreenCanvas + Worker está fora desta fase

OffscreenCanvas move trabalho para outra thread, mas não reduz o custo de desenho. O diagnóstico
mostrou que o problema é o algoritmo/pipeline de render: a main thread consegue renderizar muito
mais rápido quando o pipeline é cirúrgico. Portanto, a solução correta é **não redesenhar o que não
mudou**, antes de considerar worker.

OffscreenCanvas só volta à discussão se, depois do pipeline incremental, ainda houver jank causado
por competição real com DOM/app JS na main thread.

### Decisão 3 — Arquitetura-alvo: camadas separadas

O pipeline final deve separar a composição em três responsabilidades:

```txt
FRAME layer   (offscreen A): fundo, grid, eixos, labels, legenda
SERIES layer  (offscreen B): linhas, áreas, scatter, stacked bands
VISIBLE canvas: composição FRAME + SERIES + crosshair overlay
```

Regra de ouro:

- dados mudaram → redesenha **SERIES**;
- domínio/tamanho/opções visuais mudaram → redesenha **FRAME**;
- crosshair mudou → recompõe visible + desenha overlay, sem redesenhar séries.

## Diagnóstico atual

Hoje `ChartBase.draw()` faz:

1. Se `dirtyLayout`, chama `updateGridDomain()`.
2. Se sujo, chama `renderStatic(plot)`.
3. `renderStatic` limpa o offscreen inteiro.
4. Desenha grid.
5. Desenha eixos.
6. Desenha séries.
7. Desenha legenda.
8. `surface.blit()` copia offscreen para visible.
9. Crosshair é desenhado por cima.

O ponto crítico é que `renderStatic` mistura **moldura** e **dados**. Assim, qualquer append força:

- `generateTicks` no grid;
- `generateTicks` novamente nos axes;
- `fillText` de labels de eixo;
- `renderLegend`;
- todas as séries.

Isso é correto visualmente, mas errado para streaming. Labels e grid não precisam ser refeitos a
cada amostra.

## Escopo desta versão

### 1. Profiling interno por etapa

- [ ] Instrumentar localmente o pipeline de render para medir:
  - `updateGridDomain`;
  - `renderGrid`;
  - `renderAxes`;
  - `renderSeries` por tipo (`line`, `area`, `scatter`, `stacked`);
  - `renderLegend`;
  - composição/blit;
  - crosshair overlay.
- [ ] Medir baseline nos cenários:
  - D1: 1 gráfico, 3 séries, ring `maxPoints = 10_000`, appendFrame a 100 Hz;
  - D2: 1 gráfico, 3 séries, ring `maxPoints = 100_000`, appendFrame a 100 Hz;
  - B1: 20 gráficos, 2 séries, 2k pontos, atualização 1 Hz;
  - interação: viewport ativo com zoom/pan.
- [ ] Não commitar instrumentation permanente no runtime. Se virar ferramenta, isolar em script/dev flag.

### 2. Dirty flags granulares

Substituir o modelo atual (`dirtyLayout` / `dirtyPaint`) por flags mais precisas, mantendo a API pública.

Flags propostas:

```ts
dirtyDomain; // domínio precisa ser recomputado
dirtyFrame; // fundo/grid/eixos/labels/legenda precisam redesenhar
dirtySeries; // séries precisam redesenhar
dirtyOverlay; // crosshair/hover precisa recompor visible
```

Regras:

- `append`, `appendBatch`, `appendFrame`:
  - sempre `dirtySeries = true`;
  - `dirtyDomain = true` se a janela/extent pode mudar;
  - `dirtyFrame = true` **somente** se o domínio/ticks/legenda/tamanho mudarem de verdade.
- `setData`, `setMaxPoints`, `clear`:
  - `dirtyDomain = true`, `dirtySeries = true`;
  - `dirtyFrame = true` se domínio/ticks/labels mudarem.
- opções visuais de série:
  - `dirtySeries = true`.
- opções de eixo, padding, fonte, cores de grid/texto:
  - `dirtyFrame = true`.
- crosshair move:
  - `dirtyOverlay = true` apenas.

Compatibilidade:

- manter `invalidate('layout' | 'paint')` como fachada interna inicialmente;
- mapear a fachada antiga para as flags novas durante a migração;
- remover ou reduzir o uso semântico de `dirtyLayout` só depois dos testes passarem.

### 3. Cache de ticks e labels

Hoje grid e axes calculam ticks separadamente. Esta fase deve introduzir um cache por domínio:

```ts
interface TickCache {
  xKey: string;
  yLeftKey: string;
  yRightKey?: string;
  xTicks: number[];
  xUnit?: TimeTickUnit;
  yLeftTicks: number[];
  yRightTicks?: number[];
  xLabels: string[];
  yLeftLabels: string[];
  yRightLabels?: string[];
}
```

Regras:

- grid e axes consomem os mesmos ticks;
- labels são formatados uma vez por mudança de domínio/opção de formato;
- cache invalida quando mudam:
  - domínio X/Y;
  - `xTicks` / `yTicks`;
  - `xAxis.type`, `timeZone`, `tickFormat`;
  - `yAxis.tickFormat`;
  - fonte/tamanho se labels dependerem de medida futura.

### 4. Separação de camadas em `Surface`

Evoluir `src/render/surface.ts` de um offscreen único para duas camadas internas:

```ts
frameCanvas / frameCtx    // fundo, grid, axes, labels, legend
seriesCanvas / seriesCtx  // séries clipadas ao plot
visible ctx               // composição final + crosshair
```

Responsabilidades:

- `measure()` redimensiona visible e invalida ambas as camadas;
- contextos offscreen recebem o mesmo DPR transform;
- `destroy()` / teardown soltam referências;
- `blit()` vira composição:
  - limpar visible;
  - `drawImage(frameCanvas)`;
  - `drawImage(seriesCanvas)`;
  - overlay crosshair se ativo.

Critério não negociável:

- O clip de séries da v1.7.0 permanece na layer SERIES.
- Grid/axes continuam fora do clip.
- Crosshair continua no canvas visível para movimento barato.

### 5. Dividir `renderStatic`

Substituir o conceito monolítico por funções internas:

```ts
renderFrameLayer(plot): void
renderSeriesLayer(plot): void
composeLayers(plot): void
renderOverlay(plot): void
```

`draw()` deve virar, conceitualmente:

```ts
if (dirtyDomain) updateGridDomain();
if (dirtyFrame) renderFrameLayer(plot);
if (dirtySeries) renderSeriesLayer(plot);
composeLayers(plot);
if (showCrosshair) renderOverlay(plot);
clearDirtyFlags();
```

Mas com cuidado: `clearDirtyFlags` só pode limpar o que foi realmente redesenhado.

### 6. Streaming fast path

Criar um caminho explícito para streaming quando:

- tamanho do canvas não mudou;
- viewport não mudou;
- opções visuais não mudaram;
- domínio/ticks não mudaram;
- apenas novos pontos chegaram.

Nesse caso, por frame:

1. atualizar stores;
2. redesenhar series layer;
3. compor frame + series;
4. crosshair se ativo.

Não pode chamar:

- `renderGrid`;
- `renderAxes`;
- `formatTimeTick` / `formatNumber` para labels;
- `renderLegend`.

### 7. Testes obrigatórios

Adicionar ou atualizar testes para garantir:

- `appendFrame` sem mudança de domínio não redesenha frame layer;
- mudança de domínio redesenha frame layer;
- resize invalida frame + series;
- mudança de `textColor`, `gridColor`, fonte ou tick formatter invalida frame;
- mudança de cor/lineWidth de série invalida series;
- crosshair move não invalida frame nem series;
- viewport/zoom/pan invalida frame + series quando domínio visível muda;
- dual-Y continua desenhando eixo direito;
- stacked area continua correto;
- clip de séries continua aplicado apenas à series layer;
- `destroy()` limpa referências das duas camadas;
- DPR aplica transform nas duas camadas.

### 8. Métricas de aceite

Baseline deve ser medido antes da mudança e comparado depois.

Metas desta fase:

- Cenário D1 incremental: appendFrame + draw p50 **< 16 ms**;
- Cenário D1 incremental: p95 **< 33 ms**;
- crosshair move: p50 **< 2 ms**;
- grid/axes/labels não executam em frame de streaming sem mudança de domínio;
- bundle gzip não pode exceder o budget configurado em `size-limit`;
- nenhum crescimento contínuo de heap em teste curto de streaming.

Se D2 (100k × 3 séries) não alcançar 16 ms, ainda é aceitável se:

- D1 alcançar 16 ms;
- D2 demonstrar ganho mensurável de pelo menos 3× no p50;
- o gargalo restante estiver documentado para a próxima fase (path building / decimation cache).

## Fora de escopo

- OffscreenCanvas + Worker.
- WebGL.
- Novo tipo de gráfico.
- Subpath exports.
- CSV.
- APIs novas de streaming (`pause`, `resume`, `maxRenderRate`) — só entram depois que o render incremental estiver correto.
- Refatorar `renderLine` para abstrações bonitas. Hot loop continua com aritmética inline.

## Arquivos provavelmente afetados

- `src/render/surface.ts` — novas camadas offscreen, composição e DPR.
- `src/charts/chart-base.ts` — dirty flags, draw pipeline, divisão de render.
- `src/render/axes.ts` — consumir ticks/labels pré-resolvidos ou cacheados.
- `src/types.ts` — tipos internos para cache/camadas, se necessário.
- `src/charts/__tests__/chart-base.test.ts` — dirty flags, draw pipeline, viewport/streaming.
- `src/render/__tests__/surface.test.ts` — duas camadas + DPR.
- `src/render/__tests__/axes.test.ts` — ticks/labels compartilhados.

## Ordem de implementação recomendada

1. Criar baseline local por etapa (sem mudar runtime final).
2. Introduzir dirty flags granulares mantendo compatibilidade interna.
3. Adicionar cache de ticks/labels e adaptar grid/axes.
4. Expandir `Surface` para frame layer + series layer.
5. Dividir `renderStatic` em frame/series/compose/overlay.
6. Migrar draw pipeline para usar as novas flags.
7. Adicionar testes obrigatórios.
8. Medir D1/D2/B1 antes/depois.
9. Ajustar README/CHANGELOG com números da própria Goro, sem narrativa competitiva.

## Critérios de aceite

- [ ] `draw()` não redesenha grid/eixos/labels em frame de streaming sem mudança de domínio.
- [ ] `draw()` continua compondo crosshair sem redesenhar séries.
- [ ] Viewport, zoom, pan, dual-Y, stacked area e clipping continuam corretos.
- [ ] DPR funciona nas duas camadas.
- [ ] Testes unitários e README check passam.
- [ ] `npm run size` continua dentro do budget.
- [ ] Métricas D1 atingem p50 < 16 ms ou justificativa técnica documentada.
- [ ] Nenhuma API pública muda.
