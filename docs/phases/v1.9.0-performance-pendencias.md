# v1.9.0 — Finalização de performance da series layer

> Plano futuro de implementação. Este arquivo define como finalizar o trabalho
> de performance que ainda depende da series layer: **o que fazer**, **onde mexer**,
> **quais invariantes preservar** e **como provar que melhorou**.

## 1. Ponto de partida

O pipeline atual já tem a base necessária para otimização incremental:

- `dirtyDomain` / `dirtyFrame` / `dirtySeries` / `dirtyOverlay`;
- frame layer separada da series layer;
- series layer condicional em ring/streaming;
- cache de ticks/labels;
- fast path quando frame pode ser reaproveitado;
- crosshair/overlay fora do repaint de séries.

Essa base melhora muito:

- B1: muitos gráficos pequenos;
- viewport fixo;
- overlay/crosshair;
- casos onde grid/axes/labels não mudam.

O próximo passo é atacar o custo restante da series layer: em streaming pesado,
ela ainda é limpa e reconstruída inteira.

## 2. Problema que precisa ser resolvido

Nos cenários D1/D1-slide/D2, o custo dominante passa a ser:

1. resolver janela visível;
2. decimar/amostrar pontos visíveis;
3. reconstruir comandos/path de cada série;
4. desenhar novamente todos os segmentos visíveis.

Enquanto isso continuar acontecendo em toda mutação, a fase de performance não
pode ser considerada encerrada.

## 3. Princípio da solução

Não tentar novamente o append-only ingênuo.

O append-only simples falha quando o domínio X desliza, porque pixels antigos
ficam projetados em posições erradas. A solução precisa cachear **dados
intermediários**, não pixels finais.

A ordem correta é:

1. instrumentar o custo por etapa;
2. cachear a decimação/segmentos por série;
3. redesenhar a series layer usando segmentos cacheados;
4. só depois avaliar append-only ou translação de bitmap.

## 4. Implementação recomendada — cache de segmentos decimados

### 4.1 Objetivo

Evitar recalcular toda a decimação da janela visível a cada frame.

Em vez de cada renderer percorrer a janela inteira e decidir tudo do zero, cada
série mantém um cache de segmentos/buckets já reduzidos. Quando chegam pontos
novos, apenas os buckets afetados são recalculados.

### 4.2 Novo módulo

Criar:

```txt
src/render/decimation-cache.ts
```

Responsabilidade:

- receber `SeriesView`, domínio X visível, largura do plot e modo de gap;
- devolver uma representação compacta dos pontos/segmentos a desenhar;
- reaproveitar buckets não alterados entre frames;
- invalidar cache quando mudar algo que afete o resultado.

### 4.3 Estrutura sugerida

```ts
interface DecimatedPoint {
  logical: number;
  x: number;
  y: number;
}

interface DecimatedBucket {
  bucketIndex: number;
  logicalStart: number;
  logicalEnd: number;
  xMin: number;
  xMax: number;
  points: DecimatedPoint[];
  hasGap: boolean;
}

interface DecimationCacheKey {
  seriesId: number;
  storeVersion: number;
  xMin: number;
  xMax: number;
  plotWidth: number;
  gapMode: string;
}
```

Observação: se ainda não existir `storeVersion`, adicionar contador monotônico em
`SeriesStore` incrementado em `setData`, `append`, `appendBatch`, `appendFrame`,
`clear` e mudanças de ring. O cache não deve tentar inferir mutação por tamanho.

### 4.4 Algoritmo inicial

Para cada série:

1. resolver `[iStart, iEnd]` com `resolveRenderWindow`;
2. dividir a janela em buckets por coluna CSS ou por grupo de colunas;
3. para cada bucket:
   - se `storeVersion`, faixa lógica e parâmetros são iguais, reutilizar;
   - senão recalcular;
4. cada bucket deve guardar pontos suficientes para preservar forma visual:
   - primeiro ponto;
   - último ponto;
   - mínimo Y;
   - máximo Y;
   - pontos de quebra para `NaN` quando `gapMode='break'`;
5. concatenar buckets na ordem lógica para o renderer.

Essa estratégia mantém compatibilidade com o princípio atual de decimação por
resolução horizontal, mas evita refazer buckets que não mudaram.

### 4.5 Onde integrar

Arquivos esperados:

- `src/render/line.ts`
- `src/render/area.ts`
- `src/render/scatter.ts`
- `src/render/stacked-band.ts`
- `src/charts/chart-base.ts`
- `src/data/series-store.ts`
- `src/types.ts` se precisar expor tipo interno novo

Integração mínima:

1. `ChartBase.renderOne` cria/obtém cache por série;
2. renderer recebe uma lista decimada em vez de percorrer o `SeriesView` inteiro;
3. manter fallback atual sem cache para reduzir risco;
4. ativar cache primeiro só em ring mode (`maxPoints > 0`).

## 5. Implementação alternativa — cache de Path2D

Só considerar depois do cache de segmentos.

### Como fazer

1. Criar `src/render/path-cache.ts`;
2. cachear `Path2D` por série + domínio + plot + estilo relevante;
3. invalidar quando mudar:
   - storeVersion;
   - domínio X/Y;
   - plot rect;
   - gapMode;
   - lineWidth/dash;
   - modo stacked;
4. usar fallback quando `Path2D` não existir.

### Risco

`Path2D` pode não melhorar node-canvas e pode ter custo alto de memória. Não
implementar sem profiling mostrando que path building é gargalo relevante.

## 6. Append-only correto

Append-only só entra depois de cache intermediário.

### Permitido

Usar append-only apenas quando:

- domínio X não mudou; ou
- viewport fixo; ou
- existe mecanismo de reprojeção/translação que mantém pixels antigos corretos.

### Proibido

Não fazer:

```txt
clearRect(cauda)
redraw somente último trecho
```

em domínio deslizante. Isso deforma o gráfico.

### Caminho seguro

1. detectar `xScaleStable`;
2. se estável, desenhar apenas segmento novo;
3. se não estável, usar cache de segmentos e redraw completo da series layer;
4. só otimizar domínio deslizante depois com translação/reprojeção.

## 7. Translação de bitmap da frame layer

Baixa prioridade.

O pipeline atual já removeu grande parte do custo de frame. Só implementar se
profiling mostrar que frame layer voltou a ser relevante.

### Como fazer se necessário

1. detectar deslocamento X em pixels entre frames;
2. copiar frameCanvas deslocado para outro buffer;
3. limpar faixa exposta;
4. redesenhar apenas ticks/grid/labels da faixa;
5. validar que labels não duplicam nem ficam borradas por DPR.

Não fazer antes de resolver series layer.

## 8. Instrumentação obrigatória antes de otimizar

Criar ou expandir:

```txt
scripts/render-baseline.mjs
```

O script precisa medir separadamente:

- `updateGridDomain`;
- `tickCache.refresh`;
- `renderFrameLayer`;
- `resolveRenderWindow`;
- decimation/cache;
- path building;
- series draw;
- `surface.blit`;
- overlay/crosshair.

Resultado esperado no console:

```txt
D1-slide
  domain        p50=...
  ticks         p50=...
  frame         p50=...
  window        p50=...
  decimation    p50=...
  seriesDraw    p50=...
  blit          p50=...
  total         p50=...
```

Sem essa decomposição, não implementar otimização nova.

## 9. Testes obrigatórios

Adicionar testes antes/depois da mudança:

### Cache/invalidação

- cache reutiliza buckets quando append não afeta bucket antigo;
- cache invalida bucket afetado por append;
- `setData` invalida tudo;
- `clear` invalida tudo;
- resize/plot width invalida tudo;
- mudança de `gapMode` invalida;
- mudança de domínio Y reprojeta sem corromper dados.

### Correção visual lógica

- `NaN` com `gapMode='break'` preserva quebra;
- `gapMode='connect'` continua conectando;
- `gapMode='zero'` continua usando zero;
- dual-Y usa domínio correto;
- stacked area acumula corretamente;
- ring wrap não troca ordem lógica;
- viewport estreito mantém ponto de contexto antes/depois.

### Performance estrutural

- em append simples, número de buckets recalculados é menor que total;
- em viewport fixo, append-only só roda quando `xScaleStable=true`;
- em domínio deslizante, append-only ingênuo não é usado.

## 10. Benchmarks obrigatórios

Rodar antes/depois:

- D1: `10k × 3 séries`, ring;
- D1-slide: ring cheio, janela deslizando;
- D2: `100k × 3 séries`, ring;
- B1: `20 charts × 2 séries`;
- fast path viewport fixo;
- crosshair move.

Registrar p50, p95, p99 e média.

## 11. Critério de pronto

Só considerar performance realmente fechada quando:

- D1 p50 < 16 ms;
- D1-slide p50 < 16 ms;
- D2 tiver meta explícita realista e melhora significativa documentada;
- B1 não regredir;
- fast path continuar abaixo de 2 ms;
- bundle continuar dentro do `size-limit`;
- heap não crescer continuamente em streaming prolongado;
- nenhum documento afirmar que append-only existe se ele não existir.

## 12. Ordem de execução

1. Instrumentar `scripts/render-baseline.mjs` por etapa.
2. Rodar baseline atual e confirmar gargalo dominante.
3. Adicionar `storeVersion` em `SeriesStore`.
4. Implementar `DecimationCache` isolado.
5. Integrar primeiro em `renderLine`.
6. Medir D1/D1-slide.
7. Integrar em `renderArea` e `renderScatter`.
8. Tratar stacked area separadamente.
9. Só então avaliar append-only estável.

## 13. Fora do escopo

Não fazer nesta pendência:

- API 2.0;
- novos tipos de gráfico;
- WebGL;
- OffscreenCanvas/Worker sem profiling;
- streaming API nova (`pause`, `resume`, `maxRenderRate`);
- subpath exports;
- CSV;
- refactor amplo dos renderers sem ganho medido.
