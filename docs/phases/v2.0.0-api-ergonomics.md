# v2.0.0 — API ergonomics: options por categoria e modo de ingestão declarado

> **Origem:** visão canônica de produto + diagnóstico da superfície pública atual (`ChartOpts`).
> **Semver:** **major**. Reorganiza a superfície pública de configuração. Breaking assumido.
> **Risco:** **alto** de superfície (muitos call sites), **baixo** de lógica (não muda render/dados).
> **Prioridade:** entra **depois** da finalização de performance da series layer. Não fazer no meio
> do refino de pipeline — reescrever a superfície duas vezes seria desperdício.
> **Janela:** enquanto a base de usuários é ~zero, este é o momento certo para quebrar. Depois fica caro.

## Objetivo

Tornar a configuração da Goro Charts **impossível de usar errado por engano**, agrupando as opções
por categoria e eliminando campos concorrentes, sem alterar o comportamento de render ou de dados.

O alvo é o que bibliotecas maduras (ApexCharts, ECharts, Recharts) fazem bem: **o nome do grupo diz a
categoria**. O usuário nunca fica em dúvida se algo é dado, visual, comportamento ou tema, e nunca
existe mais de um lugar para configurar a mesma coisa.

Esta fase é de **ergonomia e tipos**, não de funcionalidade. Nenhum pixel novo é desenhado.

## Princípio central — regra de categorização

Toda opção pertence a **exatamente uma** categoria. O namespace **é** a categoria:

| Categoria                 | Namespace                            | O que contém                            |
| ------------------------- | ------------------------------------ | --------------------------------------- |
| Dados / estrutura         | `series`, `defaultSeries`, `palette` | o que é plotado e seus defaults         |
| Eixos                     | `axes.x`, `axes.y`                   | escala, ticks, formatação, domínio fixo |
| Visual / tema             | `theme`                              | cores de moldura, fontes, fundo         |
| Interação visual          | `crosshair`, `tooltip`               | mira e balão                            |
| Comportamento / streaming | `streaming`                          | janela, autoDraw, follow                |
| Dados ausentes            | `gaps`                               | política de NaN chart-wide              |
| Render (limites)          | `render`                             | thinning, limites de desenho            |
| Layout                    | `layout`                             | padding                                 |

Regras invioláveis:

1. **Nenhuma opção aparece em dois namespaces.**
2. **Override por série vive só no `SeriesConfig`** e sempre vence o default chart-wide.
3. **Uma única cadeia de precedência documentada** (ver §"Precedência").
4. O nome do campo não repete o namespace (`theme.background`, não `theme.bgColor`).

## Diagnóstico da API atual (conferido em types.ts)

O `ChartOpts` de hoje é **parcialmente aninhado, majoritariamente plano**, e mistura cinco categorias
na raiz. O que já está bom: `xAxis`, `yAxis`, `tooltip` são sub-objetos — o padrão certo. O resto
está solto.

### Problema 1 — categorias misturadas na raiz

Convivem no mesmo nível: dados (`series`, `maxPoints`, `gapMode`), comportamento (`autoDraw`,
`yMin`/`yMax`, `xTicks`/`yTicks`, `maxDots`), tema (`gridColor`, `axisColor`, `textColor`, `fontSize`,
`fontFamily`, `bgColor`, `crosshairColor`, `crosshairWidth`, `pointRadius`) e layout (`padding`).

### Problema 2 — campos concorrentes (o mais grave)

Cinco campos da raiz **duplicam** campos do `SeriesConfig`, funcionando como "fallback":

| Campo raiz    | Concorre com                    |
| ------------- | ------------------------------- |
| `lineColor`   | `SeriesConfig.color`            |
| `lineWidth`   | `SeriesConfig.lineWidth`        |
| `fillColor`   | `SeriesConfig.fillColor`        |
| `fillOpacity` | `SeriesConfig.fillOpacity`      |
| `pointColor`  | marcador de crosshair por série |

Não são redundantes (têm papel de default), mas são **concorrentes**: dois lugares para a mesma
propriedade visual, com precedência implícita. Isso é o que faz a API "parecer livre demais".

### Problema 3 — mesmo conceito em dois níveis

`yMin`/`yMax` existem na raiz (**pin do domínio do grid**) e no `SeriesConfig` (**override por
série**). Escopos diferentes, mesmo nome — fonte de confusão.

### Problema 4 — ingestão por métodos livres

`setData` / `append` / `appendBatch` / `appendFrame` são métodos livres que podem se contradizer
(`append` não faz carry-forward; misturar com `appendFrame` quebra o alinhamento). Falta um **modo
declarado** que torne o caminho certo o caminho óbvio.

## Decisões técnicas

### Decisão 1 — Namespaces por categoria, campos renomeados sem o prefixo redundante

Agrupar tudo conforme a regra de categorização. Renomear removendo prefixos que o namespace já dá.

### Decisão 2 — Matar os fallbacks concorrentes

Os cinco campos raiz (`lineColor`, `lineWidth`, `fillColor`, `fillOpacity`, `pointColor`) somem.
Substituídos por **um** `defaultSeries: Partial<SeriesConfig>` e/ou `palette: string[]`. Um lugar só
para defaults; série continua podendo sobrescrever.

### Decisão 3 — Domínio fixo mora no eixo

`yMin`/`yMax` chart-wide migram para `axes.y.min` / `axes.y.max`. O override por série permanece em
`SeriesConfig.yMin`/`yMax`, agora claramente distinto de escopo (eixo vs série).

### Decisão 4 — Modo de ingestão declarado

O modo é derivado de `streaming.maxPoints`:

- **presente** → modo _stream_; caminho canônico `appendFrame` / `appendFrames`;
- **ausente** → modo _snapshot_; caminho canônico `setData`.
  `append` / `appendBatch` viram escape hatch de baixo nível (uma série, ritmo independente),
  documentados como avançado. Tipos e mensagens de erro guiam para o caminho do modo ativo.

### Decisão 5 — Clean break, sem adaptador (recomendado)

Como a base de usuários é ~zero, a recomendação é **quebrar direto**, sem camada de compatibilidade.
Menos código, menos dívida. O adaptador flat→agrupado fica documentado como **opção** caso apareça
sinal de uso real antes do release (ver §Compatibilidade).

## Proposta — nova superfície `ChartOptions`

```ts
interface ChartOptions {
  // ── DADOS / ESTRUTURA ────────────────────────────────
  series: SeriesConfig[];
  /** Defaults aplicados a toda série que não sobrescrever. Substitui os
   *  antigos lineColor / lineWidth / fillColor / fillOpacity soltos na raiz. */
  defaultSeries?: SeriesDefaults;
  /** Cores atribuídas por índice quando a série não define `color`. */
  palette?: string[];

  // ── EIXOS ────────────────────────────────────────────
  axes?: {
    x?: XAxisConfig; // type, tickFormat, timeZone, ticks
    y?: YAxisConfig; // tickFormat, ticks, min, max (pin de domínio)
  };

  // ── VISUAL / TEMA ────────────────────────────────────
  theme?: {
    background?: string; // era bgColor
    text?: string; // era textColor
    axis?: string; // era axisColor
    grid?: string; // era gridColor
    font?: { size?: number; family?: string }; // era fontSize / fontFamily
  };

  // ── INTERAÇÃO VISUAL ─────────────────────────────────
  crosshair?: {
    color?: string; // era crosshairColor
    width?: number; // era crosshairWidth
    markerColor?: string; // era pointColor
    markerRadius?: number; // era pointRadius (confirmar escopo — ver nota)
  };
  tooltip?: TooltipConfig; // inalterado (xFormat, valueFormat)

  // ── COMPORTAMENTO / STREAMING ────────────────────────
  streaming?: {
    maxPoints?: number; // era maxPoints (raiz)
    autoDraw?: boolean; // era autoDraw (raiz)
    followLatest?: boolean; // novo comportamento declarado de streaming
  };

  // ── DADOS AUSENTES ───────────────────────────────────
  gaps?: GapMode; // era gapMode (default chart-wide)

  // ── RENDER (limites) ─────────────────────────────────
  render?: {
    maxDots?: number; // era maxDots (scatter thinning)
  };

  // ── LAYOUT ───────────────────────────────────────────
  layout?: {
    padding?: [number, number, number, number]; // era padding (raiz)
  };
}
```

Ajustes nos sub-tipos:

```ts
interface XAxisConfig {
  type?: ScaleType;
  tickFormat?: (value: number) => string;
  timeZone?: string;
  ticks?: number; // era ChartOpts.xTicks
}

interface YAxisConfig {
  tickFormat?: (value: number) => string;
  ticks?: number; // era ChartOpts.yTicks
  min?: number; // era ChartOpts.yMin (pin de domínio; undefined = auto)
  max?: number; // era ChartOpts.yMax
}

/** Defaults de série; qualquer campo visual do SeriesConfig, sem id/name. */
type SeriesDefaults = Partial<Omit<SeriesConfig, 'id' | 'name'>>;
```

`SeriesConfig` permanece a fonte de verdade por série (id, name, color, lineWidth, dash, fillColor,
fillOpacity, yAxis, stack, yMin, yMax, hidden, valueFormat, gapMode). Nada muda nele além de agora ser
o **único** lugar de override.

> **Nota a resolver na implementação:** confirmar se `pointRadius` de hoje controla só o marcador do
> crosshair ou também o tamanho do ponto no `ScatterChart`. Se controla os dois, separar em
> `crosshair.markerRadius` e um `defaultSeries.dotRadius` (ou campo equivalente no ScatterChart),
> para não recriar um campo com dois papéis.

`defaultSeries`.

## Precedência (uma cadeia, documentada)

Para qualquer propriedade visual de série, o valor resolvido é:

```
SeriesConfig[i].<campo>  ??  defaultSeries.<campo>  ??  palette (só para color)  ??  built-in default
```

Para domínio Y:

```
SeriesConfig[i].yMin/yMax  (override por série)
  ??  axes.y.min / axes.y.max  (pin do grid)
  ??  auto (derivado dos dados)
```

Nunca mais de duas origens para a mesma coisa, e a ordem é sempre esta. Documentar explicitamente no
JSDoc de cada campo com override.

## Modo de ingestão declarado

```ts
// modo snapshot (sem streaming.maxPoints)
const chart = new LineChart(canvas, { series: [...] });
chart.setData('cpu', xs, ys);            // caminho canônico

// modo stream (com streaming.maxPoints)
const chart = new LineChart(canvas, {
  series: [...],
  streaming: { maxPoints: 10_000, followLatest: true },
});
chart.appendFrame(t, { cpu, mem });      // caminho canônico
chart.appendFrames(batch);               // backfill/replay

// escape hatch de baixo nível (documentado como avançado)
chart.append('cpu', t, v);               // série única, ritmo independente
```

Regras:

- Em modo snapshot, chamar `append*` lança erro descritivo apontando para `setData` (comportamento
  atual já lança "ring mode not active" — manter e melhorar a mensagem).
- Em modo stream, `setData` continua válido para semear histórico inicial antes de streamar.
- A documentação apresenta `appendFrame`/`appendFrames` como padrão de streaming; `append`/`appendBatch`
  como caso de séries de timestamp independente.

## Tipos que previnem erro

- **Namespaces** já impedem a maior parte dos enganos (categoria explícita).
- `defaultSeries` como `Partial<Omit<SeriesConfig,'id'|'name'>>` — impossível dar id/name a um default.
- `axes.y.min`/`max` como `number | undefined` — `undefined` é o sentinela de "auto"; `0` é bound
  legítimo (preserva a correção do v1.x sobre `yMin: 0`).
- Considerar **factory tipada** por modo, se valer:
  `createStreamingChart(canvas, opts)` vs `createStaticChart(canvas, opts)`, cada um expondo só os
  métodos do seu modo. Opcional — avaliar custo/benefício na implementação; não obrigatório para o v2.

## Compatibilidade

**Recomendado: clean break.** Sem adaptador. Publica-se o v2 com a superfície nova, `CHANGELOG` com
guia de migração e tabela de mapeamento (a de cima). Justificativa: base de usuários ~zero; adaptador
é dívida sem retorno.

**Opção (só se surgir sinal de uso antes do release):** adaptador `flat → grouped`.

```ts
// utilitário de migração, um minor de vida, com deprecation warning
function upgradeOptions(old: LegacyChartOpts): ChartOptions;
```

- aceita o `ChartOpts` plano, retorna o `ChartOptions` agrupado;
- emite `console.warn` de deprecação uma vez por instância;
- removido no v2.1.0.

Decidir clean break vs adaptador **antes** de começar, para não instrumentar os dois caminhos.

## Nota de CHANGELOG / semver

`major`:

> **Breaking:** configuration surface reorganized by concern. Flat options grouped into
> `series`/`defaultSeries`/`axes`/`theme`/`crosshair`/`streaming`/`gaps`/`render`/`layout`.
> Competing per-series fallbacks removed in favor of `defaultSeries` + `palette`. Fixed Y domain
> moved to `axes.y.min/max`. Ingestion mode is now derived from `streaming.maxPoints`. See migration
> table. No behavioral change to rendering or data.
